# 每日台股當沖推薦報告 — 雲端自動生成版

你在一個全新的雲端沙盒環境執行，**沒有瀏覽器、沒有 Playwright**。只能用 Bash（curl、node）、WebFetch、WebSearch。這份文件已經內建完整方法論，不用參考其他地方。

執行完直接把結果寫進本 repo 的 `data/latest.json`，然後 commit + push 到 `main` branch，不用問任何人、不用解釋過程。

**最高原則：資料品質比看起來完整更重要。** 任何一步如果你無法確認資料完整或正確（例如抓不到全部分頁、查不到某檔股票的處置股狀態、國際盤勢時間點有時差疑慮），就把該項寫進 `data_quality.warnings`（見下方 JSON 格式），並在對應的選股/結論上誠實降級信心度，不要假裝資料完整。寧可交出一份「標了警示但誠實」的報告，也不要交出一份「看起來完整但其實漏東西」的報告。

## 第 1 步：抓成交量排行

前往 TAIFEX 行情網（mis.taifex.com.tw）「一般交易時段行情 → 股票類契約報價」，篩選到期月份為近月合約。

**這個網站的已知陷阱**（本機用 Playwright 手動操作時踩過）：
- 頁面右下角「熱門股票期貨」快速連結區塊的清單**跟真正的成交量排行不是同一份**，不要用。
- 網站的表格「點擊排序」只在當前已載入的那一頁內排序，不是跨全部資料排序。近月合約通常有 297 筆、分 3 頁（每頁 100 筆），排在後面分頁的高量股會被漏掉。

**你在雲端沒有瀏覽器，做法改成**：
1. 用 `curl` 直接抓這個頁面（或它背後實際回傳資料的 API/AJAX 端點——用 `curl -v` 先看回應是完整 HTML 表格還是需要額外 POST 參數）。如果是 server-side render 的 HTML 表格，直接用 `node` 寫一段腳本解析 HTML（正則或簡單字串處理抓「商品」「成交量」欄位），不要交給 WebFetch 的摘要模型（大量表格資料容易被摘要模型漏抓或幻覺，之前在解析 Yahoo Finance 分時資料時就出過這個問題）。
2. 如果頁面需要分頁參數（例如 `?page=2`），逐頁抓取，並且**確認你抓到的總筆數/總頁數，跟頁面上顯示的「共 N 筆」或「第 X 頁，共 Y 頁」一致**。不一致就是抓漏了，要繼續抓或在 warnings 註明「TAIFEX 成交量排行可能未抓取完整（只取得 X/Y 頁）」。
3. 如果 curl 抓不到可用資料（例如整頁都是 JS 渲染、curl 拿到空殼），退而求其次用 WebFetch 抓這個頁面，但**明確要求 WebFetch 逐行列出所有看到的「商品」與「成交量」，不要摘要**，並且事後核對數量是否合理（近月通常 250-300 檔）；數量明顯偏少（例如只有十幾檔）就代表被摘要模型截斷了，要在 warnings 註明「成交量排行透過 WebFetch 取得，可能不完整，建議人工複核」。

## 第 2 步：去重、轉換成股票

第一頁商品名稱是「XX期貨」，把同一支股票的重複月份合約去重，去掉「期貨」兩字就是對應股票代號。

用 WebSearch 或直接查證券交易所公開資訊（例如查「[股票代號] 收盤價」）取得每支股票現貨的收盤價、漲跌幅、振幅、開高低——這是股票本身的資料，不是期貨合約的資料。

## 第 3 步：篩選

依漲跌幅「絕對值」排序（不分正負），拿掉波動最小的 2-3 支。

## 第 4 步：排行與分組

剩下的股票依波動幅度排序給排行，每一名附入選理由。檢查是否有龍頭股（例如聯電之於晶圓代工），有的話把同產業/供應鏈相關族群整理在一起分組呈現。

## 第 5 步：每檔給完整交易計畫

對每一檔給下一交易日的五項（缺一不可）：進場價、停利點、目標價、停損點、策略說明。依前一日開高低收與振幅計算。若該股已經漲停鎖住，要特別標注追價風險。

**每檔都要給 A 計畫（拉回進場）／B 計畫（開盤沒拉回、直接站上前高噴出時的備案）**：
- **A 計畫**：三段式風控——進場區間 → 早期減碼點（跌破就先出一半）→ 最終停損（跌破就全出）。早期減碼點設在進場區間下緣與最終停損之間。
- **B 計畫**：突破追價用「站穩前高不破一段時間才追價＋收緊停損」的邏輯，單一收緊停損，不疊加二段式。
- 明確講清楚什麼情況下 A、B 計畫都不成立就直接放棄不做，寫進 `summary` 文字。

## 第 6 步：兩層風險體檢（缺一不可，這是本流程最容易出錯也最重要的一步）

**a. 處置股／注意股體檢**
每一檔都要用 WebSearch 查是否被 TWSE/櫃買中心列為注意股或處置股（搜尋「[股票名稱] 處置股 注意股 [年月]」）。
- 處置股：改成人工撮合，等於無法做正常當沖 —— **直接排除**。
- 注意股：風險偏高但還能交易，要在該檔的 `risk_tag` 標注。
- 查不到明確資訊的股票，不要假設它是乾淨的——在 warnings 標注「[股票代號] 處置股狀態未能確認」，並在該檔的信心度上保守處理。

**b. 國際盤勢背景**
查當天/前一天美股（道瓊、那斯達克、標普）收盤，以及亞股（日經、KOSPI，尤其牽涉半導體/記憶體時查三星、SK 海力士）走勢。

**時差陷阱（務必檢查）**：美股開盤是台灣時間晚上 9:30、收盤是隔天凌晨 4 點左右。這份報告是台灣時間早上 8:30 產生，**這個時間點美股當晚那一場理論上已經收盤**（凌晨 4 點 vs 8:30，已經過了），所以正常情況下不會有時差問題。但仍然要做一件事：**查詢時明確確認你查到的美股數字是「收盤價」不是「盤中報價」**——查證時注意新聞或數據來源的時間戳記，如果查到的資訊時間戳落在台灣時間凌晨 4 點之前（代表還沒收盤），要換一個來源重新查，不要把盤中價當收盤價。查不到明確收盤數據時，在 warnings 標注「國際盤勢資料可能非最終收盤值」。

## 第 7 步：新聞：濃縮重點＋多面向延伸分析

用 WebSearch 找當天相關新聞（則數不限），給：濃縮重點、多面向延伸分析（不只股票影響，也要有國際趨勢/地緣政治/總經角度）。

## 第 8 步：報告結構

**結論給兩檔**：一檔「安全牌」（風險最低、最穩）、一檔「衝最快」（波動最大、積極型），各自附完整 A/B 計畫。挑「衝最快」時，同樣要先過處置股/注意股體檢，優先選沒被處置過的高波動股。

若第 6 步的風險體檢顯示風險偏高，結論要誠實降級為「保守單」，不要為了給信心而硬撐高信心度。

## 第 9 步：輸出格式

把結果寫成這個 JSON 結構，覆寫 `data/latest.json`（**注意：欄位名稱要完全照這個格式，前端 `index.html` 是照這個格式 render 的**）：

## 第 10 步：commit 並 push

```bash
git config user.name "daily-report-bot"
git config user.email "daily-report-bot@users.noreply.github.com"
git add data/latest.json
git commit -m "Daily report for $(date +%Y-%m-%d)"
git push
```

如果 push 失敗（例如遠端有其他 commit），先 `git pull --rebase` 再 push 一次。

## 交付前最後檢查

- 有沒有把盤中報價當收盤價？
- 每一檔選股是否都做過處置股/注意股查證？
- TAIFEX 成交量排行是否確認頁數/筆數完整？
- 以上任何一項有疑慮，`data_quality.warnings` 是否都誠實列出來了？

檢查完直接 commit push，不用回報過程。
